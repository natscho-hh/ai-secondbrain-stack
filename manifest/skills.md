# Skill Manifest

A curated catalog of third-party skills that work well with this template's vault workflows. This is a starting point, not a closed list — Phase 4 of `SETUP.md` reads this file, proposes matches against what the user said they want to do, and also searches community directories for anything not covered here. Every entry below was checked against its real, public source at the time of writing; none are invented or recalled from memory alone.

None of these are installed by the template itself. They are third-party skills — install only one at a time, only after running the security gate from `SETUP.md` Phase 4, and only with the user's explicit OK for that specific skill.

## Where skills go

The **Install** column below shows each tool's own **upstream** command — the destination that tool's own docs recommend, which is often a single agent's private config directory (for example Claude Code's `~/.claude/skills`, or wherever `npx skills add` / a given CLI's `init` command defaults to). Followed literally, a Codex or Gemini CLI user would install the skill somewhere their own agent never looks, breaking the portability this template promises.

For a skill to be portable across agents, it needs to live in the **vault's own `skills/` folder** — the same place the template's core skills live, and the folder every agent is told to check via `AGENTS.md`'s skill reflex rule. So: run the upstream Install command as shown (it's still the correct, canonical way to fetch the skill), but then place or copy the resulting skill folder into the vault's `skills/<name>/` — instead of, or in addition to, wherever the tool put it by default. That one extra step is what makes the skill visible to every agent, not just the one whose default path the installer happened to target.

## Discovery

This table is not exhaustive, and skill ecosystems move fast — repos get renamed, superseded, or abandoned. When a user's stated goal (from Phase 2) doesn't match anything above:

1. **Search the community directories**, not training-data memory:
   - [github.com/travisvn/awesome-claude-skills](https://github.com/travisvn/awesome-claude-skills) — curated list, browse by category
   - [awesomeclaudeskills.com](https://awesomeclaudeskills.com) — searchable directory
   - A direct web search for `"<goal keyword>" claude skill github` also works well — skill authors tend to name repos descriptively.
2. **Surface 2-3 candidates** to the user with a one-line purpose each, rather than picking one unilaterally.
3. **Never install from a search result on the strength of its description alone.** Before proposing any third-party skill — one found here in the table or one found fresh via search — run the exact security gate defined in `SETUP.md` Phase 4:
   > Before installing, I check: (1) Is the source a public repo with real usage? (2) I read the skill's files for commands that touch things outside the vault (network calls, deletions, credential access). (3) I summarize what it does in one sentence and wait for your OK.
4. **Install only after the user says OK** for that specific skill. If the gate raises any concern — private or unknown source, no real usage, commands that reach outside the vault without a clear reason — say so plainly and let the user decide with that information in hand.
5. If a URL in this manifest turns out to be dead, renamed, or replaced by the time someone reads it, treat it as unverified, search fresh, and update this file rather than installing from a broken link.
